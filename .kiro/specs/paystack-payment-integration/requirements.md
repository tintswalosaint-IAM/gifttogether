# Requirements Document

## Introduction

This feature integrates Paystack as the payment processor for the NEO gift registry platform (formerly GiftTogether). It enables contributors to make real payments towards gift goals, with funds split automatically between the creator's bank account (via a Paystack subaccount) and the NEO platform account. Creators register their bank details once; contributors pay a gross-up amount that covers the NEO service fee and Paystack transaction fees, ensuring the creator always receives the exact contribution amount they intended.

The integration covers: creator bank account onboarding, a bank list lookup endpoint, payment initialisation with fee gross-up, payment verification, webhook-based confirmation, and the full mobile payment UI flow (pre-payment summary → Paystack hosted checkout → processing → success).

---

## Glossary

- **NEO**: The gift registry platform (formerly GiftTogether). The system under specification.
- **Creator**: An authenticated NEO user who owns a registry and receives contributions.
- **Contributor**: Any person (authenticated or guest) who pays towards a gift goal.
- **Paystack**: The South African payment processor used by NEO.
- **Subaccount**: A Paystack entity that represents a creator's bank account, identified by a `subaccount_code`. Paystack uses it to route the creator's share of each payment directly to their bank.
- **Subaccount_Code**: The unique string identifier returned by Paystack when a subaccount is created (e.g. `ACCT_xxxxxxxxxxxxxxx`).
- **NEO_Fee**: The flat R2.00 platform service fee charged to the contributor per transaction.
- **Paystack_Fee**: The Paystack transaction fee of 2.9% of the charged amount plus R1.00, deducted by Paystack from the gross charge.
- **Contribution_Amount**: The rand value the contributor intends to send to the creator (e.g. R200.00). The creator receives this amount in full.
- **Gross_Amount**: The total amount charged to the contributor's card, calculated to ensure the creator receives the full Contribution_Amount after all fees are deducted. Formula: `Gross_Amount = (Contribution_Amount + NEO_Fee + R1.00) / (1 - 0.029)`, rounded up to the nearest cent.
- **Fee_Breakdown**: The itemised display shown to the contributor before payment: Contribution_Amount + NEO_Fee = Total charged.
- **Payment_Reference**: A unique string generated by NEO for each payment initialisation, used to correlate Paystack callbacks and webhooks with the local transaction record.
- **Access_Code**: The short-lived token returned by Paystack's `POST /transaction/initialize` endpoint, used to open the hosted checkout page at `https://checkout.paystack.com/{access_code}`.
- **Authorization_URL**: The full Paystack-hosted checkout URL returned alongside the Access_Code.
- **Webhook**: An HTTP POST sent by Paystack to NEO's `POST /api/payments/webhook` endpoint when a payment event occurs (e.g. `charge.success`).
- **Webhook_Secret**: The Paystack webhook secret used to verify the HMAC-SHA512 signature on incoming webhook requests. Stored in `appsettings.json` under `Paystack:WebhookSecret`.
- **Pending_Payment**: A local database record created when a payment is initialised, tracking the reference, goal ID, contributor details, and amounts before Paystack confirms the charge.
- **Contribution**: The existing NEO database entity (`Contribution` model) that records a confirmed payment against a `GiftGoal`.
- **Payment_Controller**: The ASP.NET Core controller at route `api/payments` that handles all payment-related endpoints.
- **PaystackService**: The backend service class responsible for all HTTP communication with the Paystack API.
- **BankSetup_Screen**: The mobile UI screen where a creator enters their bank name, account number, and account holder name.
- **PaymentSummary_Screen**: The mobile UI screen shown to a contributor before redirecting to Paystack, displaying the fee breakdown and a "Pay" button.
- **Processing_Screen**: The mobile UI screen shown after returning from Paystack checkout, displaying an animated checklist while verification runs.
- **Success_Screen**: The mobile UI screen shown after successful payment verification, displaying "You're part of this! 💛" and an updated progress bar.

## Requirements

### Requirement 1: Paystack Configuration

**User Story:** As a platform operator, I want Paystack credentials stored securely in configuration, so that secret keys are never hardcoded in source code.

#### Acceptance Criteria

1. THE NEO SHALL read the Paystack secret key from `appsettings.json` under the key `Paystack:SecretKey`.
2. THE NEO SHALL read the Paystack webhook secret from `appsettings.json` under the key `Paystack:WebhookSecret`.
3. THE NEO SHALL read the NEO platform subaccount code (for receiving the R2 fee) from `appsettings.json` under the key `Paystack:PlatformSubaccountCode`.
4. IF the `Paystack:SecretKey` configuration value is absent or empty at application startup, THEN THE NEO SHALL throw a configuration exception and refuse to start.
5. THE PaystackService SHALL include the secret key as a Bearer token in the `Authorization` header of every outbound Paystack API request.

---

### Requirement 2: Creator Bank Account Registration

**User Story:** As a creator, I want to register my bank account details with NEO, so that contributions are paid directly into my bank account via Paystack.

#### Acceptance Criteria

1. WHEN a creator submits their bank name, account number, and account holder name via `POST /api/payments/register-bank`, THE Payment_Controller SHALL call Paystack `POST /subaccount` to create a subaccount.
2. WHEN Paystack returns a `subaccount_code` in response to the subaccount creation call, THE Payment_Controller SHALL persist the `subaccount_code` on the creator's `User` record in the database.
3. THE NEO SHALL set the Paystack subaccount `percentage_charge` to `0` when creating a subaccount, because the split is defined per-transaction using a flat amount rather than a percentage.
4. IF the creator already has a `subaccount_code` stored on their `User` record, THEN THE Payment_Controller SHALL call Paystack `PUT /subaccount/{subaccount_code}` to update the existing subaccount rather than creating a duplicate.
5. IF Paystack returns an error during subaccount creation or update, THEN THE Payment_Controller SHALL return HTTP 502 with a descriptive error message and SHALL NOT persist any changes to the `User` record.
6. THE `POST /api/payments/register-bank` endpoint SHALL require a valid Bearer token and return HTTP 401 if the token is absent or invalid.
7. IF the bank name, account number, or account holder name field is empty or whitespace, THEN THE Payment_Controller SHALL return HTTP 400 with a descriptive validation error before calling Paystack.

---

### Requirement 3: Bank List Lookup

**User Story:** As a contributor or creator, I want to select my bank from a dropdown populated with real South African bank names, so that I don't have to type bank names manually.

#### Acceptance Criteria

1. WHEN a client calls `GET /api/payments/banks`, THE Payment_Controller SHALL call Paystack `GET /bank?currency=ZAR` and return the list of banks as a JSON array.
2. THE Payment_Controller SHALL return each bank entry with at minimum the bank `name` and `code` fields from the Paystack response.
3. THE Payment_Controller SHALL cache the bank list response in memory for 24 hours to avoid redundant Paystack API calls.
4. IF Paystack returns an error on the bank list call, THEN THE Payment_Controller SHALL return HTTP 502 with a descriptive error message.
5. THE `GET /api/payments/banks` endpoint SHALL NOT require authentication, so that the bank list is accessible during the creator onboarding flow before login.

---

### Requirement 4: Fee Gross-Up Calculation

**User Story:** As a contributor, I want the total amount I pay to cover all fees, so that the creator always receives the exact contribution amount they intended.

#### Acceptance Criteria

1. THE NEO SHALL calculate the Gross_Amount using the formula: `Gross_Amount = ceil((Contribution_Amount + NEO_Fee + 1.00) / (1 - 0.029) * 100) / 100`, where NEO_Fee is R2.00 and the result is rounded up to the nearest cent.
2. WHEN a contributor intends to contribute R200.00, THE NEO SHALL calculate a Gross_Amount of R208.80 (rounded up to the nearest cent).
3. THE NEO SHALL calculate the creator's split amount as equal to the Contribution_Amount (e.g. R200.00 for a R200.00 contribution).
4. THE NEO SHALL calculate the NEO platform split amount as R2.00 per transaction.
5. THE NEO SHALL pass the creator's split amount to Paystack as the `subaccount` bearer amount in the transaction initialisation, so Paystack routes the Contribution_Amount to the creator's subaccount and the remainder (after Paystack fees) to the NEO platform account.

---

### Requirement 5: Payment Initialisation

**User Story:** As a contributor, I want to initiate a payment for a specific gift goal, so that I can be redirected to Paystack's secure checkout page.

#### Acceptance Criteria

1. WHEN a client calls `POST /api/payments/initialize` with a `goalId` and `contributionAmount`, THE Payment_Controller SHALL validate that the goal exists and that `contributionAmount` is greater than zero.
2. WHEN the goal and amount are valid, THE Payment_Controller SHALL calculate the Gross_Amount using the fee gross-up formula defined in Requirement 4.
3. THE Payment_Controller SHALL generate a unique Payment_Reference for each initialisation request using the format `NEO-{goalId}-{timestamp}-{random6chars}`.
4. THE Payment_Controller SHALL create a Pending_Payment record in the database before calling Paystack, storing the Payment_Reference, goalId, contributionAmount, grossAmount, contributor name (if provided), and contributor message (if provided).
5. THE Payment_Controller SHALL call Paystack `POST /transaction/initialize` with: `amount` set to Gross_Amount in kobo (Gross_Amount × 100, as an integer), `reference` set to the Payment_Reference, `currency` set to `ZAR`, `subaccount` set to the creator's Subaccount_Code, `bearer` set to `subaccount`, `transaction_charge` set to Contribution_Amount in kobo, and `callback_url` set to the NEO deep link callback URL.
6. WHEN Paystack returns a successful response, THE Payment_Controller SHALL return HTTP 200 with the `access_code`, `authorization_url`, and `reference` to the client.
7. IF the creator does not have a `subaccount_code` registered, THEN THE Payment_Controller SHALL return HTTP 422 with the error "Creator has not registered a bank account for payouts."
8. IF Paystack returns an error during initialisation, THEN THE Payment_Controller SHALL return HTTP 502 with a descriptive error message and SHALL mark the Pending_Payment record as failed.
9. THE `POST /api/payments/initialize` endpoint SHALL NOT require authentication, so that guest contributors can pay without creating an account.

---

### Requirement 6: Payment Verification

**User Story:** As a contributor, I want the app to verify my payment after returning from Paystack checkout, so that my contribution is recorded and the goal progress is updated.

#### Acceptance Criteria

1. WHEN a client calls `POST /api/payments/verify/{reference}`, THE Payment_Controller SHALL call Paystack `GET /transaction/verify/{reference}` to confirm the transaction status.
2. WHEN Paystack confirms the transaction status is `success`, THE Payment_Controller SHALL look up the corresponding Pending_Payment record by reference.
3. WHEN the Pending_Payment record is found and not yet confirmed, THE Payment_Controller SHALL create a `Contribution` record with the `ContributorName`, `Message`, `Amount` set to the Contribution_Amount (not the Gross_Amount), and `GiftGoalId`.
4. WHEN the Contribution record is created, THE Payment_Controller SHALL mark the Pending_Payment record as confirmed so that duplicate verification calls do not create duplicate Contribution records.
5. WHEN the Contribution record is created, THE Payment_Controller SHALL return HTTP 200 with the contribution details and the updated `TotalRaised` for the goal.
6. IF the Paystack transaction status is not `success`, THEN THE Payment_Controller SHALL return HTTP 402 with the error "Payment was not successful."
7. IF the Payment_Reference does not match any Pending_Payment record, THEN THE Payment_Controller SHALL return HTTP 404 with the error "Payment reference not found."
8. IF the Pending_Payment record is already confirmed, THEN THE Payment_Controller SHALL return HTTP 200 with the existing contribution details (idempotent response).
9. THE `POST /api/payments/verify/{reference}` endpoint SHALL NOT require authentication.

---

### Requirement 7: Paystack Webhook Handler

**User Story:** As a platform operator, I want NEO to handle Paystack webhooks, so that payments are confirmed reliably even if the contributor closes the app before the verify call completes.

#### Acceptance Criteria

1. WHEN Paystack sends a POST request to `POST /api/payments/webhook`, THE Payment_Controller SHALL read the raw request body and the `x-paystack-signature` header.
2. THE Payment_Controller SHALL compute the HMAC-SHA512 of the raw request body using the Webhook_Secret and compare it to the `x-paystack-signature` header value.
3. IF the computed signature does not match the header value, THEN THE Payment_Controller SHALL return HTTP 400 and SHALL NOT process the event.
4. WHEN the signature is valid and the event type is `charge.success`, THE Payment_Controller SHALL extract the `reference` from the event data and attempt to confirm the corresponding Pending_Payment using the same logic as Requirement 6 (create Contribution if not already confirmed).
5. WHEN the webhook is processed successfully, THE Payment_Controller SHALL return HTTP 200 immediately, regardless of whether the Pending_Payment was already confirmed.
6. WHEN the event type is not `charge.success`, THE Payment_Controller SHALL return HTTP 200 without taking any action (acknowledge and ignore unknown events).
7. THE Payment_Controller SHALL NOT call Paystack's verify API from within the webhook handler; it SHALL trust the webhook payload directly after signature verification.

---

### Requirement 8: Creator Bank Account Setup — Mobile UI

**User Story:** As a creator, I want a mobile screen to enter my bank account details, so that I can register my payout account without leaving the app.

#### Acceptance Criteria

1. THE BankSetup_Screen SHALL display a dropdown field populated with the bank list from `GET /api/payments/banks`, labelled "Bank name".
2. THE BankSetup_Screen SHALL display a text input for the account number, labelled "Account number", accepting numeric input only.
3. THE BankSetup_Screen SHALL display a text input for the account holder name, labelled "Account holder name".
4. THE BankSetup_Screen SHALL display a "Save bank account" button that is disabled while a submission is in progress.
5. WHEN the creator taps "Save bank account" with all fields filled, THE BankSetup_Screen SHALL call `POST /api/payments/register-bank` and display a success message on completion.
6. IF any required field is empty when the creator taps "Save bank account", THEN THE BankSetup_Screen SHALL display an inline validation error beneath the empty field and SHALL NOT submit the form.
7. IF the API call returns an error, THEN THE BankSetup_Screen SHALL display the error message to the creator without navigating away.
8. THE BankSetup_Screen SHALL be accessible from the creator's profile settings screen and from the collection creation flow.
9. WHEN the bank list is loading, THE BankSetup_Screen SHALL display a loading indicator in the bank dropdown.

---

### Requirement 9: Contributor Payment Summary — Mobile UI

**User Story:** As a contributor, I want to see a clear fee breakdown before I pay, so that I understand exactly what I will be charged.

#### Acceptance Criteria

1. WHEN a contributor enters a contribution amount and taps "Pay", THE PaymentSummary_Screen SHALL display the goal name, the Contribution_Amount, the NEO_Fee (R2.00), and the total amount to be charged (Contribution_Amount + R2.00).
2. THE PaymentSummary_Screen SHALL display the label "Service fee" next to the R2.00 NEO_Fee line item.
3. THE PaymentSummary_Screen SHALL display a "Pay R{total}" button where `{total}` is the sum of Contribution_Amount and R2.00 (e.g. "Pay R202.00" for a R200.00 contribution).
4. WHEN the contributor taps the "Pay R{total}" button, THE PaymentSummary_Screen SHALL call `POST /api/payments/initialize` and, on success, open the Authorization_URL in the device browser or an in-app WebView.
5. WHILE the initialisation API call is in progress, THE PaymentSummary_Screen SHALL disable the pay button and display a loading indicator.
6. IF the initialisation API call returns an error, THEN THE PaymentSummary_Screen SHALL display the error message and re-enable the pay button.
7. THE PaymentSummary_Screen SHALL display a note that the payment is secured by Paystack.

---

### Requirement 10: Payment Processing — Mobile UI

**User Story:** As a contributor, I want to see a processing screen after returning from Paystack checkout, so that I know the app is verifying my payment.

#### Acceptance Criteria

1. WHEN the contributor returns to the app from the Paystack checkout (via deep link callback), THE NEO SHALL navigate to the Processing_Screen.
2. THE Processing_Screen SHALL display an animated checklist with the items: "Secure payment", "Verifying details", "Almost there…" appearing sequentially with a short delay between each item.
3. WHILE the Processing_Screen is visible, THE NEO SHALL call `POST /api/payments/verify/{reference}` in the background using the Payment_Reference from the deep link callback.
4. WHEN verification succeeds, THE NEO SHALL navigate from the Processing_Screen to the Success_Screen.
5. IF verification fails, THE NEO SHALL navigate from the Processing_Screen to an error state displaying the failure reason and a "Try again" button that re-triggers verification.
6. THE Processing_Screen SHALL display the animated checklist for a minimum of 1.5 seconds before navigating away, even if verification completes faster, to avoid a jarring instant transition.

---

### Requirement 11: Payment Success — Mobile UI

**User Story:** As a contributor, I want to see a success screen after my payment is confirmed, so that I know my contribution was received.

#### Acceptance Criteria

1. THE Success_Screen SHALL display the message "You're part of this! 💛".
2. THE Success_Screen SHALL display the updated progress bar for the gift goal, reflecting the new TotalRaised amount returned by the verify endpoint.
3. THE Success_Screen SHALL display a "Share this collection" button that opens the native share sheet with the registry URL.
4. THE Success_Screen SHALL display a "Back to collection" button that navigates the contributor back to the registry page.
5. WHEN the contributor taps "Back to collection", THE NEO SHALL navigate to the registry page and refresh the goal list to show the updated TotalRaised.

---

### Requirement 12: Deep Link Callback Handling

**User Story:** As a contributor, I want the app to automatically return from the Paystack browser and begin verification, so that I don't have to manually navigate back.

#### Acceptance Criteria

1. THE NEO SHALL register a deep link URI scheme (e.g. `neo://payment/callback`) that Paystack uses as the `callback_url` in the transaction initialisation.
2. WHEN the device OS invokes the deep link URI after Paystack checkout, THE NEO SHALL extract the `reference` query parameter from the URI.
3. WHEN the `reference` parameter is present in the deep link URI, THE NEO SHALL navigate to the Processing_Screen and begin payment verification.
4. IF the `reference` parameter is absent from the deep link URI, THEN THE NEO SHALL navigate to the registry page and display an error message "Payment reference missing. Please check your contribution history."
5. THE NEO SHALL handle the deep link on both Android and iOS (MAUI Blazor Hybrid platforms).

---

### Requirement 13: Pending Payment Data Model

**User Story:** As a platform operator, I want pending payments tracked in the database, so that webhook and verify calls can reliably correlate with the original payment intent.

#### Acceptance Criteria

1. THE NEO SHALL store a `PendingPayment` entity in the database with the fields: `Id`, `Reference` (unique string), `GiftGoalId`, `ContributionAmount` (decimal), `GrossAmount` (decimal), `ContributorName` (nullable string), `ContributorMessage` (nullable string), `Status` (enum: Pending, Confirmed, Failed), and `CreatedAt` (UTC datetime).
2. THE NEO SHALL enforce a unique index on `PendingPayment.Reference`.
3. WHEN a Contribution record is created from a PendingPayment, THE NEO SHALL store the `PendingPaymentId` as a foreign key on the `Contribution` record to link the two.
4. THE NEO SHALL store the `PaystackSubaccountCode` on the `User` model as a nullable string field.
